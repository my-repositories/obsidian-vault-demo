# Лучшие практики и типичные ошибки GitHub SpecKit

## Общие принципы работы

### ✅ Лучшие практики

1. **SpecKit — не волшебная пилюля, а инструмент**
   - Работает только при правильном использовании
   - Требует понимания процесса и внимательности
   - Вы контролируете AI, а не наоборот

2. **Спецификации — живые документы**
   - Обновляйте при каждом изменении требований
   - Не бойтесь дополнять по мере роста проекта
   - AI следует актуальным документам

3. **Критическое мышление на каждом шаге**
   - Не принимайте на веру сгенерированный AI
   - Задавайте вопросы, просите объяснений
   - Запрашивайте альтернативы для ключевых решений

4. **Пошаговый контроль качества**
   - Тестируйте после каждой фазы implement
   - Коммитьте после каждой фазы
   - Делайте `/clear` между сессиями

### ❌ Общие ошибки

1. **Слишком много надежд на AI**
   - AI не заменит ваше понимание проекта
   - Без чётких указаний AI фантазирует
   - SpecKit помогает, но не решает всё за вас

2. **Попытки ускорить процесс пропуском шагов**
   - Пропуск clarify → дыры в спецификации
   - Пропуск analyze → переписывание кода
   - Пропуск тестирования → накопление багов

3. **Работа в одной сессии над всеми фазами**
   - Контекстное окно переполняется
   - Качество кода падает
   - Увеличиваются галлюцинации
   - Тратятся токены впустую

## Constitution

### ✅ Best Practices

**Начните с минимума:**
```
1. Python FastAPI + Vue.js + TypeScript
2. Telegram Bot API для публикации
3. Markdown конвертер с проверкой синтаксиса
4. Токены хранить в переменных окружения
5. Один коммит = одна таска
```

**Обновляйте по мере роста:**
Когда сталкиваетесь с проблемами — добавляйте новые принципы

**Используйте существующий код:**
```bash
/speckit.constitution
Проанализируй текущий код и создай constitution на его основе.
```

**Лайфхак: генерация через ChatGPT**
Опишите ваши требования ChatGPT и попросите создать первый драфт constitution

### ❌ Типичные ошибки

**Слишком общие формулировки:**
```
❌ "Код должен быть качественным"
✅ "Функции не длиннее 30 строк, обязательная типизация TypeScript"
```

**Перебор с тестами:**
```
❌ "100% coverage для всех функций" (потратите все токены)
✅ "Unit тесты для бизнес-логики, integration для API endpoints"
```

## Specify

### ✅ Best Practices

**Пишите для человека:**
Ваш коллега без знания проекта должен понять что вы хотите построить

**Будьте максимально подробны:**
Чем детальнее опишите — тем меньше вопросов и переделок потом

**Используйте модальные глаголы:**
```
Система ДОЛЖНА:
 - Сохранять черновики автоматически каждые 30 секунд
 - Показывать ошибки форматирования понятным языком

Система МОЖЕТ:
 - Предлагать исправление типичных ошибок markdown

Система НЕ ДОЛЖНА:
 - Публиковать без явного подтверждения от пользователя
```

**Разделяйте MVP и будущее:**
```
MVP:
- Markdown редактор с превью
- Публикация в один канал
- Автосохранение

Версия 2.0:
- Публикация в несколько каналов
- Шаблоны постов
```

### ❌ Типичные ошибки

**Технические детали в spec:**
```
❌ "Используй FastAPI для backend, PostgreSQL для хранения"
✅ "Пользователи сохраняют черновики. Система автоматически сохраняет изменения"
```

**Слишком общее описание:**
```
❌ "Приложение для постинга в Telegram с удобным интерфейсом"
✅ "Пользователь пишет текст, видит превью как в Telegram, публикует одним кликом"
```

**Нечёткие критерии:**
```
❌ "Приложение должно быть быстрым"
✅ "Превью обновляется < 100ms, публикация < 2 секунд"
```

## Clarify

### ✅ Best Practices

**Отвечайте подробно:**
```
❌ "B) Показать ошибку"
✅ "Показывать предупреждение в редакторе. Предложить варианты:
   1. Сократить вручную
   2. Разбить на несколько постов автоматически
   3. Сохранить как черновик"
```

**Уделяйте внимание безопасности:**
- Валидация данных
- Хранение токенов
- Права доступа
- Rate limiting
- XSS/CSRF защита

**Исследуйте если не знаете:**
Потратьте 5 минут на гугление, посмотрите как делают конкуренты

### ❌ Типичные ошибки

**Пропустить clarify:**
```
❌ "У меня и так всё продумано" → дыры обнаруживаются в разработке
✅ Всегда запускайте clarify, даже если уверены что всё учли
```

**Ответы "не знаю" или "потом решим":**
```
❌ AI придумает всё на свой вкус (часто неправильно)
✅ Исследуйте вопрос и дайте обоснованный ответ
```

**Игнорировать "скучные" вопросы про безопасность:**
```
❌ "Потом добавлю безопасность"
✅ Безопасность нельзя "потом добавить"
```

## Checklist

### ✅ Best Practices

**Минимум 2 чек-листа:**
- UX — для всех проектов
- Security — для всех проектов с пользовательским вводом

**Честно используйте:**
1. Создайте чек-лист
2. Честно ответьте на каждый пункт
3. Для каждого ❌ дополните спецификацию
4. Перезапустите и проверьте что все ✅

**Создайте "Master Checklist":**
После нескольких проектов накопите опыт и создайте один большой чек-лист с:
- Самыми частыми проблемами
- Вашими слабыми местами (что обычно забываете)
- Спецификой вашей ниши

### ❌ Типичные ошибки

**Создать и не использовать:**
```
❌ Создать чек-лист, пробежаться глазами и идти дальше
✅ Честно пройти по каждому пункту и дополнить spec для каждого ❌
```

## Plan

### ✅ Best Practices

**Читайте критически:**
- Почему выбран именно этот фреймворк?
- Нужна ли такая сложная архитектура?
- Можно ли проще?

**Запрашивайте альтернативы:**
```
Ты предложил PostgreSQL для БД. Какие альтернативы есть?
Сравни PostgreSQL vs SQLite vs MongoDB для этого проекта.
Плюсы/минусы каждого варианта.
```

**Обновляйте при изменениях:**
Каждое изменение стека/архитектуры → обновление plan.md

**Используйте как справочник:**
- Забыли структуру API? → Открыли "API Specification"
- Не помните как называется таблица? → Открыли "Database Schema"

### ❌ Типичные ошибки

**Принять план не глядя:**
```
❌ AI предложил → принял → пошёл кодить
✅ Внимательно изучить минимум Technology Stack и Architecture
```

**Не обновлять при изменениях:**
```
❌ Изменили стек в процессе, plan.md остался старым
✅ Каждое изменение → обновление plan.md
```

**Лайфхак: ChatGPT для review:**
```
Вот технический план проекта: [plan.md]

Проанализируй:
1. Есть ли переусложнения?
2. Где могут быть bottleneck'и?
3. Какие риски ты видишь?
4. Что можно упростить для MVP?
```

## Tasks

### ✅ Best Practices

**Работайте строго по фазам:**
- Не прыгайте между фазами
- Закончили Phase 1 полностью → переходите к Phase 2
- Каждая фаза — логически завершённый блок

**Каждая задача = один коммит:**
```bash
git commit -m "T005: Create Markdown Editor Component

- Add editor with syntax highlighting
- Configure live preview
- Implement auto-save

Closes T005"
```

**Следуйте Dependencies:**
- Проверяйте Dependencies перед началом задачи
- Если хотите сделать задачу вне очереди — убедитесь что все dependencies выполнены

### ❌ Типичные ошибки

**Игнорировать Dependencies:**
```
❌ Пытаетесь сделать T022 (публикация в Telegram), не сделав T012 (Bot API)
✅ Проверяйте зависимости перед началом
```

**Не обновлять tasks.md:**
```
❌ Изменили архитектуру, tasks.md остался старым
✅ Обновляйте соответствующие задачи при изменениях
```

**Лайфхак: разбивайте сложные задачи:**
```
Задача T014 слишком сложная. Разбей её на 5-7 под-задач,
каждая должна быть выполнима за 30-60 минут.
```

## Analyze

### ✅ Best Practices

**Запускайте минимум 2 раза:**
1. После создания всех спецификаций
2. После исправления критичных проблем

**Все Critical — исправлять ДО разработки:**
Critical issues названы так не просто — они блокируют функциональность

**Запускайте повторно до чистого результата:**
После исправления проблем могут появиться новые

**Просите объяснений для непонятных проблем:**
```
Issue H001 — не понял.
Объясни на примере: что конкретно может пойти не так?
Приведи сценарий шаг за шагом.
```

**Просите предложить решения:**
```
Как лучше всего исправить C003?
Предложи 3 варианта с плюсами и минусами каждого.
```

### ❌ Типичные ошибки

**Пропустить analyze:**
```
❌ Сразу идти в implement → проблемы в процессе разработки
✅ Analyze — обязательный шаг перед implement
```

**Игнорировать Critical:**
```
❌ "Ну это мелочь, потом исправлю"
✅ Все Critical — исправлять до начала разработки
```

**Исправлять "на глаз" без обновления документов:**
```
❌ Решили мысленно делать по-другому, но не обновили spec/plan
✅ Каждое исправление → обновление файла → повторный analyze
```

## Implement

### ✅ Best Practices

**Одна фаза = одна сессия чата:**
```
Сессия 1:
1. /speckit.implement Phase 1
2. Ждёте завершения
3. Тестируете результат
4. Коммитите
5. /clear

Сессия 2:
1. /speckit.implement Phase 2
... и так далее
```

**Тестируйте после каждой фазы:**
- Баг в Phase 1 может сломать Phase 2, 3, 4
- Найдёте только в конце и придётся всё переделывать

**Читайте сгенерированный код:**
- Хотя бы бегло
- Если непонятно — просите объяснений

**Коммитьте после каждой фазы:**
- Если что-то сломается — сможете откатиться к рабочему коммиту
- AI может получить код из предыдущего коммита если что-то пошло не так

### ❌ Типичные ошибки

**Запустить без analyze:**
```
❌ Implement → проблемы → переделываем код
✅ Analyze → исправить → implement
```

**Всё за одну сессию:**
```
❌ Переполнение контекста, галлюцинации, плохой код
✅ По фазам, между фазами /clear
```

**Не тестировать после каждой фазы:**
```
❌ "Сделаю все 6 фаз, потом протестирую"
✅ Phase 1 → тест → Phase 2 → тест
```

**Принимать код не глядя:**
```
❌ "Нейросеть сгенерировала — значит работает"
✅ Всегда читайте код, просите исправить проблемы
```

## Когда использовать SpecKit

### ✅ Идеально подходит

- **Небольшие и средние проекты** (до 50 файлов)
- **Проекты где важна чистая архитектура**
- **Solo-разработчики**
- **Обучение** (понять как строить проекты правильно)

### ❌ Избыточен

- **Быстрые прототипы** ("просто показать идею")
- **Очень маленькие проекты** (1-5 файлов)
- **Где скорость > качества**

### ⚠️ Проблематичен

- **Большие legacy проекты** (нужно создавать спеки для всего)
- **Очень сложные проекты** (1000+ файлов)

## Ключевые выводы из практического опыта

1. **Clarify** — не пропускайте. Она находит то, о чём вы не подумали.
2. **Checklist** — перестраховка. UX, Security — обязательны.
3. **Plan** — критически проверяйте. Не принимайте на веру, упрощайте для MVP.
4. **Tasks** — фокус на Critical. 70% задач — это nice-to-have, а не MVP.
5. **Analyze** — спасает от переделок. Лучше потратить час на исправление спеков, чем день на переписывание кода.
6. **Implement** — работайте по фазам. Одна фаза = одна сессия. Между этапами обязательно делайте `/clear`.
7. **Тестируйте результаты работы** после каждой фазы.
