# GitHub SpecKit: вайб-кодинг на основе спецификаций

## Что это такое

**GitHub SpecKit** — инструмент от GitHub, который превращает хаотичные промпты при разработке в чёткие структурированные документы для AI-агентов. Вместо немедленной генерации кода (с галлюцинациями и ошибками) вы сначала создаёте детальные спецификации, которые становятся "источником правды" для нейросети.

**Ключевая идея:** AI-агенты меньше фантазируют, когда имеют чёткий контекст. SpecKit заставляет нейросеть понимать **ЧТО** и **ЗАЧЕМ** строить, прежде чем писать код.

## Почему это важно

При обычном вайб-кодинге (vibe coding) нейросеть начинает фантазировать/галлюцинировать и гонять по кругу ошибки вперемешку с несуществующими методами. SpecKit решает эту проблему через:

- Структурированный подход к спецификации требований
- Разделение "что строить" (spec) и "как строить" (plan)
- Автоматический поиск проблем до начала разработки
- Пошаговую реализацию с контролем качества

## Ключевые возможности

### 8 команд SpecKit

1. **`/speckit.constitution`** — создаёт "ДНК проекта" (принципы архитектурыры, стандарты кода, тестирование, git workflow, security)
2. **`/speckit.specify`** — описывает ЧТО и ЗАЧЕМ строить (без технических деталей)
3. **`/speckit.clarify`** — AI задаёт уточняющие вопросы по слепым зонам спецификации
4. **`/speckit.checklist`** — создаёт чек-листы качества (UX, Security, Performance, Testing и др.)
5. **`/speckit.plan`** — превращает спецификацию в технический план (стек, архитектура, БД, API)
6. **`/speckit.tasks`** — разбивает план на последовательные задачи с приоритетами и dependencies
7. **`/speckit.analyze`** — AI-аудит всех документов на конфликты и несоответствия
8. **`/speckit.implement`** — последовательная реализация задач из плана

### Структура файлов проекта

```
your-project/
├── .specify/
│   ├── memory/
│   │   └── constitution.md
│   ├── scripts/
│   └── templates/
├── .claude/
│   └── commands/
└── specs/
    └── 001-feature-name/
        ├── spec.md
        ├── clarify.md
        ├── plan.md
        ├── tasks.md
        ├── analysis.md
        ├── research.md
        ├── database-schema.md
        ├── api-documentation.yaml
        └── quickstart.md
```

## Как применять

### Установка (Windows)

```powershell
# Шаг 1: Установите UV
powershell -ExecutionPolicy ByPass -c "irm https://astral.sh/uv/install.ps1 | iex"

# Шаг 2: Перезапустите терминал и проверьте
uv --version

# Шаг 3: Инициализируйте проект
uvx --from git+https://github.com/github/spec-kit.git specify init .

# Шаг 4: Выберите AI-агента и тип скриптов в диалоге
```

### Рекомендуемый рабочий процесс

1. **Constitution** → Начните с 5 ключевых принципов, обновляйте по мере роста
2. **Specify** → Детально опишите WHAT/WHY, избегайте HOW
3. **Clarify** → Ответьте на вопросы AI подробно, не "не знаю"
4. **Checklist** → Минимум UX и Security, пройдите честно по каждому пункту
5. **Plan** → Укажите стек явно, критически изучите результат, упростите для MVP
6. **Tasks** → Определите critical-задачи (⭐), работайте строго по фазам
7. **Analyze** → Запустите минимум 2 раза, исправьте все Critical проблемы
8. **Implement** → Одна фаза = одна сессия чата = тест → коммит → `/clear`

**Критически важно:**
- Не запускать implement без analyze
- Работать по фазам, не всё за одну сессию
- Тестировать после каждой фазы
- Делать `/clear` между фазами для уменьшения галлюцинаций

## Инструменты и подходы

- **AI-агенты:** Cursor, Claude Code, GitHub Copilot
- **Установка:** UV (package manager для Python)
- **Методология:** Spec-Driven Development (разработка через спецификации)
- **Принципы:** SOLID, DRY, KISS, Feature-Sliced Design, Clean Architecture

## Распространённые ошибки и ограничения

### Когда SpecKit НЕ подходит
- Быстрые прототипы ("просто показать идею")
- Очень маленькие проекты (1-5 файлов)
- Где скорость важнее качества
- Большие legacy проекты (нужно создавать спеки для всего)
- Очень сложные проекты (1000+ файлов)

### Идеально для
- Небольших и средних проектов (до 50 файлов)
- Проектов где важна чистая архитектура
- Solo-разработчиков
- Обучения (понимание как строить проекты правильно)

### Типичные ошибки
- Технические детали в `/speckit.specify` (вместо "How" — только "What")
- Пропуск `/speckit.clarify` — дыры обнаруживаются уже в разработке
- Ответы "не знаю" на вопросы clarify — AI придумает сам
- Пропуск analyze → проблемы в процессе написания кода
- Запуск implement на всё сразу → переполнение контекста, галлюцинации
- Отсутствие тестирования после каждой фазы
