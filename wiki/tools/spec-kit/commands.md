# Справочник по командам GitHub SpecKit

## 1. `/speckit.constitution` — ДНК проекта

**Файл:** `.specify/memory/constitution.md`

Самый важный файл. Набор принципов, от которых AI не должен отступать.

### Что включать

- **Архитектурные принципы:** Feature-Sliced Design, Clean Architecture, Microservices
- **Технические ограничения:** Языки/версии, обязательные инструменты (ESLint, Prettier), запрещённые библиотеки
- **Стандарты качества кода:** SOLID, DRY, KISS, стандарты именования, максимальная длина функций
- **Тестирование:** Обязательные типы тестов, coverage требования, TDD подход
- **Git и коммиты:** Conventional commits, стратегия бранчей, требования к PR
- **UX/Accessibility:** WCAG стандарты, responsive design, Core Web Vitals
- **Security и Privacy:** Требования безопасности, GDPR compliance

### Best Practices

- Начните с минимума (5 принципов), добавляйте по мере проблем
- Для существующих проектов: "Проанализируй текущий код и создай constitution на его основе"
- Constitution — живой документ, обновляйте при необходимости

### Типичные ошибки

- ❌ Слишком общие формулировки: "Код должен быть качественным" (что это значит для AI?)
- ❌ Перебор с тестами: можно потратить все токены

## 2. `/speckit.specify` — Что нужно построить

**Файл:** `specs/001-feature-name/spec.md`

Описывает **ЧТО** и **ЗАЧЕМ**, игнорируя технические детали. Думайте как продуктовый менеджер.

### Главный принцип: "What" и "Why", никакого "How"

✅ **Правильно:**
```
Пользователь пишет текст в редакторе, видит превью для Telegram
и публикует одним кликом. Система автоматически конвертирует
в markdown формат Telegram и исправляет ошибки форматирования.
```

❌ **Неправильно:**
```
Используй python-telegram-bot для API, храни черновики в Redis,
парси markdown через mistune, публикуй через sendMessage endpoint.
```

Технические детали — в `/speckit.plan`!

### Структура spec.md

- **Feature Overview:** Краткое описание фичи
- **User Stories:** "Как [роль], я хочу [действие], чтобы [цель]"
- **Functional Requirements:** Конкретные измеримые требования
- **Success Criteria:** Условия успешной реализации
- **Key Entities:** Важные объекты данных
- **Assumptions:** Предварительные условия и ограничения

### Best Practices

- Пишите для человека, не для машины (коллега должен понять без знания проекта)
- Будьте максимально подробны
- Используйте "ДОЛЖНА/МОЖЕТ/НЕ ДОЛЖНА":
  ```
  Система ДОЛЖНА:
   - Сохранять черновики автоматически каждые 30 секунд
  
  Система МОЖЕТ:
   - Предлагать исправление типичных ошибок markdown
  
  Система НЕ ДОЛЖНА:
   - Публиковать без явного подтверждения от пользователя
  ```
- Разделяйте MVP и будущие фичи

### Типичные ошибки

- ❌ Технические детали в specify
- ❌ Слишком общее описание: "Приложение для постинга в Telegram с удобным интерфейсом"
- ❌ Нечёткие критерии: "Приложение должно быть быстрым" (что такое быстро?)

✅ **Правильно:**
```
- Превью обновляется < 100ms после ввода символа
- Публикация поста < 2 секунд
- Автосохранение каждые 30 секунд
- Retry при ошибке API: 3 попытки с интервалом 5 секунд
```

## 3. `/speckit.clarify` — Уточнение требований

**Файл:** `specs/001-feature-name/clarify.md`

AI анализирует спецификацию и задаёт вопросы, о которых вы сами не подумали.

### Категории вопросов

1. Authentication & Authorization
2. Data Validation & Limits
3. Error Handling
4. Edge Cases
5. Performance & Scale
6. Security & Privacy
7. API Integration

### Best Practices

- **Отвечайте подробно:**
  - ❌ Плохо: "B) Показать ошибку"
  - ✅ Хорошо: "Показывать предупреждение в редакторе. Предложить варианты: 1) Сократить вручную, 2) Разбить на несколько постов, 3) Сохранить как черновик"
- Не отвечайте "не знаю" или "потом решим" — AI придумает сам (часто неправильно)
- Уделяйте внимание вопросам безопасности

### Типичные ошибки

- ❌ Пропустить clarify ("у меня и так всё продумано") → дыры в спецификации обнаруживаются в разработке
- ❌ Игнорировать "скучные" вопросы про безопасность

## 4. `/speckit.checklist` — Чек-листы качества

**Файлы:** `specs/001-feature-name/checklists/[тема]-checklist.md`

Структурированная проверка по конкретным критериям качества.

### Доступные темы

```bash
/speckit.checklist UX
/speckit.checklist Security
/speckit.checklist Performance
/speckit.checklist Accessibility
/speckit.checklist SEO
/speckit.checklist Testing
/speckit.checklist Mobile-First Design  # кастомная тема
```

### Best Practices

- Создавайте минимум 2 чек-листа: UX и Security (для всех проектов)
- Честно отвечайте на каждый пункт
- Для каждого ❌ дополняйте спецификацию
- Перезапускайте checklist и убеждайтесь что теперь все ✅

### Аналогия

- **Clarify** — это интервью с умным коллегой
- **Checklist** — это аудит по стандартам качества

## 5. `/speckit.plan` — Технический план

**Файлы:** 
- `specs/001-feature-name/plan.md` — детальный план
- `specs/001-feature-name/research.md` — исследование технологий
- `specs/001-feature-name/database-schema.md` — схема БД
- `specs/001-feature-name/api-documentation.yaml` — OpenAPI спецификация
- `specs/001-feature-name/quickstart.md` — инструкции для старта

Мост между "что нужно построить" и "какой код писать".

### Использование

**С указанием стека (рекомендуется):**
```bash
/speckit.plan

Backend: Python FastAPI + TypeScript
Frontend: Vue.js 3
UI: Tailwind CSS + shadcn-vue
База данных: PostgreSQL
Деплой: Docker
```

**Без указания стека:**
AI сам предложит оптимальный стек на основе спецификации

### Структура plan.md

- **Technology Stack:** Детальное описание с обоснованием
- **Project Architecture:** Структура проекта с директориями
- **Database Schema:** Подробная схема с индексами
- **API Endpoints:** Детальная спецификация API
- **Implementation Details:** Специфичные детали реализации

### Best Practices

- Читайте план критически, не принимайте на веру
- Запрашивайте альтернативы для ключевых решений: "Ты предложил PostgreSQL. Какие альтернативы? Сравни PostgreSQL vs SQLite vs MongoDB"
- Обновляйте plan.md при каждом изменении стека/архитектуры

## 6. `/speckit.tasks` — Разбивка на задачи

**Файл:** `specs/001-feature-name/tasks.md`

Превращает огромный технический план в последовательность мелких выполнимых задач.

### Структура задач.md

- Сгруппировано по фазам разработки (обычно 5-7 фаз)
- Каждая задача содержит:
  - **ID** (T001, T002...)
  - **Маркер Critical** ⭐ (must-have для MVP)
  - **Сложность** (простая/средняя/сложная)
  - **Время** (оценка)
  - **Описание**
  - **Acceptance Criteria**
  - **Files to create**
  - **Dependencies**

### Best Practices

- Работайте строго по фазам, не прыгайте
- Каждая задача = один коммит
- Проверяйте Dependencies перед началом задачи
- Обновляйте tasks.md при изменении подхода

### Типичные ошибки

- ❌ Игнорировать Dependencies (пытаетесь сделать T022, не сделав T012)
- ❌ Не обновлять tasks.md при изменении архитектуры

## 7. `/speckit.analyze` — Поиск проблем

**Файл:** `specs/001-feature-name/analysis.md`

AI-аудит всех созданных документов. Как опытный тимлид, который ревьюит проект перед стартом.

### Что анализируется

- spec.md, plan.md, tasks.md
- database-schema.md, api-documentation.yaml
- constitution.md

### Типичные ошибки

- ❌ **Пропустить analyze и сразу идти в implement** → проблемы обнаруживаются в процессе разработки
- ❌ **Игнорировать Critical проблемы** → они названы так не просто, блокируют функциональность
- ❌ **Исправлять проблемы "на глаз", не обновляя документы** → AI будет следовать старым документам
- ❌ **Не запускать analyze повторно после правок** → могут появиться новые проблемы

### Best Practices

- Запускайте analyze минимум 2 раза (после создания спеков и после исправлений)
- Все Critical — исправлять **до** начала разработки
- Запускайте повторно до "No critical issues found"

## 8. `/speckit.implement` — Реализация

Финальная команда. Берёт все спецификации и начинает писать код.

### Варианты использования

**Всё сразу (не рекомендуется):**
```bash
/speckit.implement
```
Проблема: контекстное окно переполняется, качество кода падает

**По фазам (рекомендуется):**
```bash
/speckit.implement Phase 1
/speckit.implement Phase 1-2
```

**Конкретные задачи:**
```bash
/speckit.implement T001, T002, T003
```

### Рекомендуемый workflow

1. Новая сессия чата
2. `/speckit.implement Phase 1`
3. Ждёте завершения
4. Тестируете результат
5. Коммитите
6. Закрываете сессию (`/clear`)
7. Новая сессия → `/speckit.implement Phase 2`

### Best Practices

- Одна фаза = одна сессия чата
- Не пытайтесь делать несколько фаз в одной сессии (контекст загрязняется)
- Тестируйте после каждой фазы
- Коммитьте после каждой фазы
- Делайте `/clear` между этапами для уменьшения галлюцинаций и экономии токенов

### Типичные ошибки

- ❌ Запустить implement без analyze
- ❌ Пытаться сделать всё за одну сессию → переполнение контекста, галлюцинации
- ❌ Не тестировать результаты после каждой фазы
- ❌ Принимать код не глядя
- ❌ Не коммитить после каждой фазы
