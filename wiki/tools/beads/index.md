# Beads: Система памяти для кодинг-агентов

## Что это такое

**Beads** (сокращение от *bug database*) — легковесный, агенто-ориентированный трекер задач/багов, управляемый через CLI утилиту `bd`. Решает фундаментальную проблему кодинг-агентов (Claude Code, Sourcegraph Amp, OpenAI Codex и др.) — **«деменцию сессий»**.

**Ключевая проблема:** Контекстное окно AI-агентов ограничено (~10 минут активной работы), после перезапуска память стирается. Агенты создают вложенные markdown-планы без связей, теряют исходные фазы, рекурсивно плодят новые планы. При нехватке контекста агенты «срезают углы»: игнорируют сломанные тесты, отключают проверки, маркируют незавершённую работу как готовую.

## Почему это важно

### Проблемы без Beads

- **Потеря контекста:** При перезапуске агент забывает что делал
- **Markdown-планы не масштабируются:** Агенты создают вложенные планы без связей, теряют фазы
- **Игнорирование багов:** Обнаруженные баги не фиксируются, а игнорируются для экономии токенов
- **Дублирование работы:** Несколько агентов могут делать одно и то же

### Как Beads решает

- **Постоянная память:** JSONL-файлы под управлением Git — память сохраняется между сессиями
- **Структурированные данные:** Агенты запрашивают JSON, а не парсят текст
- **Граф зависимостей:** Задачи связаны через parent/child, blocking, discovered-from
- **Мульти-агентная координация:** Агенты проверяют статус задач через `bd` перед стартом

## Как это работает

### Архитектура хранения

**Формат:** JSONL-файлы (каждая строка — JSON объект)
**Управление:** Git (контроль версий, слияние изменений)
**Отказ от:** PostgreSQL (нет нужды в СУБД), Temporal (избыточен для десктопных инструментов)

### Схема данных

Минималистичная, но строго типизированная:

- **`parent/child`** — указатели для эпиков и подзадач
- **`blocking`** — блокирующие задачи
- **4 типа зависимостей:** включая `discovered-from` (для работы, найденной во время выполнения другой задачи)
- **Статусы:** `open`, `in_progress`, `ready`, `blocked`, `closed`

### Распределённость

Несколько агентов/воркеров на разных машинах работают с одной базой через общий Git-репозиторий. Конфликты слияния (включая коллизии ID на разных ветках) разрешаются ИИ через интеллектуальный merge.

## Ключевые команды CLI

### `bd quickstart`
Инициализация проекта и автоматический перенос открытых задач из markdown/TODO-листов в Beads.

### `bd ready --json`
Запрос списка разблокированных задач, готовых к выполнению. Агенты получают структурированные данные без парсинга текста.

### `--assignee <agent_id>`
Фильтрация задач по исполнителю для мульти-агентной координации. Предотвращает дублирование работы.

### Аудит изменений
Каждое изменение логируется с меткой времени и идентификатором агента. История статусов доступна без `git blame`.

## Практическое внедрение

### Шаг 1: Отказ от markdown-планов
Перенести все текущие рабочие элементы в Beads. Markdown подходит только для документации, не для оркестрации.

### Шаг 2: Декомпозиция задач
Дробить работу на мелкие `bd`-тикеты. **Каждая сессия агента выполняет ровно одну задачу.**

**Преимущества:**
- Снижает стоимость вычислений (квадратичная экономия токенов)
- Убирает паническое «закрытие» задач при лимите контекста
- Делает сессии одноразовыми и предсказуемыми

### Шаг 3: Интеграция в промпты
Добавить в `AGENTS.md` или `CLAUDE.md` инструкцию: использовать `bd quickstart` и автоматически заводить тикеты на любые обнаруженные проблемы (включая сломанные тесты или техдолг).

### Шаг 4: Мульти-агентная работа
Агенты перед стартом проверяют `status: in_progress` через `bd`, что исключает дублирование. Разные ветки и машины синхронизируются через Git.

### Шаг 5: Настройка
Установка утилиты `bd` → запуск `bd quickstart` против репозитория → агенты автоматически переключаются на тикетную модель.

## Архитектурные уроки и ограничения

### Почему не Markdown
- Markdown для агентов — **память только для записи** (write-only)
- Требует ручного копирования контекста или парсинга текста
- В Beads зависимости — **first-class citizens**
- Запрос `bd ready --json` мгновенно выдаёт пул доступных задач без интерпретации прозы

### Почему не Temporal
- Temporal избыточен для легковесных десктопных инструментов оркестрации агентов

### Почему не TypeScript для оркестраторов
- Строгая типизация не мешает ИИ писать посредственный код
- Ошибки архитектуры проявляются поздно
- Автор переписывает проект `vibecoder` на **Go**

### Текущий статус
- **Beads находится в стадии alpha**, но уже демонстрирует стабильную работу в production-сценариях вайб-кодинга

## Преимущества графа зависимостей

Граф зависимостей в Beads отражает **реальный ход разработки**, а не статичный список:

- **Поле `discovered-from`:** Связывает баги с задачами, в которых они были найдены
- **Динамическое обновление:** Новые задачи появляются в процессе выполнения существующих
- **Блокировки:** Явное указание какие задачи блокируют другие

**Аналогия:** Возврат к markdown после Beads сравнивается с попыткой удержать длинный телефонный номер в памяти без записи.
