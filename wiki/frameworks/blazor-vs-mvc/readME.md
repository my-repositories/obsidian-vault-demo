# ASP.NET: Blazor vs MVC

## Что это такое

Два подхода к веб-разработке в экосистеме ASP.NET:

- **MVC (Model-View-Controller)** — классическая архитектура с серверным рендерингом HTML. Браузер отправляет HTTP-запрос → контроллер обрабатывает → сервер возвращает полноценный HTML-документ.
- **Blazor** — современная технология для создания SPA-приложений на C# без JavaScript. C#-код выполняется либо на сервере (Blazor Server), либо в браузере через WebAssembly (Blazor WebAssembly).

## Почему это важно

Blazor **снижает порог входа** для C#-разработчиков в веб: не нужно учить JavaScript для клиентской логики. Однако MVC остаётся проверенным решением для классических многостраничных сайтов.

Понимание различий помогает выбрать правильный подход под задачу:
- **Бизнес-приложения с CRUD** → Blazor Server
- **Классические сайты/многостраничники** → MVC
- **Автономные клиентские приложения** → Blazor WebAssembly
- **High-load/real-time графика** → MVC + JavaScript (Blazor не подходит)

## Ключевые идеи

### ASP.NET MVC

**Архитектура:**
- **Model** — данные и бизнес-логика
- **View** — HTML-представление (использует Razor-синтаксис)
- **Controller** — обработка запросов, выбор модели/представления

**Как работает:**
1. Браузер отправляет HTTP-запрос
2. Контроллер вызывает модель
3. Выбирается представление (View)
4. Сервер возвращает полноценный HTML-документ (замена всей страницы)
5. Дополнительно: JavaScript + AJAX для частичного обновления страниц

**Плюсы:**
- Проверенная временем архитектура (MVC с 1970-х, Smalltalk)
- Чёткое разделение ответственности
- Гибкость и устойчивость
- Подходит для SEO (серверный рендеринг)

**Минусы:**
- Полная перезагрузка страницы (без AJAX)
- Требуется JavaScript для интерактивности
- Меньшая отзывчивость UI

### Blazor Server

**Как работает:**
1. Браузер устанавливает постоянное соединение через **SignalR** (WebSocket)
2. События UI отправляются на сервер
3. Сервер выполняет логику на C#
4. Браузер получает **дельта-изменения DOM** в реальном времени
5. Встроенный JS-адаптер фреймворка (разработчику не нужно писать JS)

**Аналогия:** Blazor Server работает как React с виртуальным DOM, но синхронизация с сервером происходит через WebSocket.

**Плюсы:**
- Вся логика на C# (без JavaScript)
- Реальное время обновления UI
- Компонентная архитектура (как React)
- Быстрая разработка для C#-разработчиков

**Минусы:**
- Постоянный поток сообщений клиент ↔ сервер
- Не подходит для игр, визуализаций с высокой частотой обновлений
- Зависимость от соединения с сервером
- Нагрузка на сервер при большом количестве клиентов

### Blazor WebAssembly (WASM)

**Как работает:**
1. На клиент загружаются: скомпилированные DLL приложения, .NET Runtime, зависимости
2. Приложение выполняется **локально в браузере**
3. Соединение SignalR — локальное (внутри браузера)
4. Последующие посещения кешируются

**Плюсы:**
- Работает автономно (без постоянного соединения)
- Минимальная нагрузка на сервер
- Поддержка всеми современными браузерами
- Лёгкая отладка
- Песочница браузера (изоляция)

**Минусы:**
- Медленный первый запуск (загрузка DLL + runtime)
- **Нет доступа к серверным ресурсам**
- Код клиента можно **декомпилировать** (не хранить секреты!)
- Клиентская логика ограничена интерфейсом

## Сравнительная таблица

| Критерий | MVC | Blazor Server | Blazor WebAssembly |
|---|---|---|---|
| **Язык клиентской логики** | JavaScript | C# (сервер) | C# (браузер) |
| **Соединение** | HTTP-запросы | SignalR (WebSocket) | Локальное (в браузере) |
| **Обновление UI** | Полная страница / AJAX | Дельта DOM | Дельта DOM |
| **Нагрузка на сервер** | Низкая | Средняя/высокая | Минимальная |
| **Первый запуск** | Быстрый | Быстрый | Медленный (загрузка DLL) |
| **SEO** | Отличное | Среднее (SSR в .NET 8+) | Плохое |
| **Безопасность** | Серверная логика | Серверная логика | Клиентская (декомпиляция) |
| **Подходит для** | Многостраничники, SEO | CRUD, бизнес-приложения | Автономные PWA |

## Как применять

### Шаг 1: Определи тип проекта

- **Многостраничный сайт, SEO важен** → MVC
- **Бизнес-приложение с формами/таблицами** → Blazor Server
- **Автономное PWA-приложение** → Blazor WebAssembly
- **Игры, real-time графика** → MVC + JavaScript

### Шаг 2: Выбери модель рендеринга (Blazor)

В .NET 8+ Blazor поддерживает смешанные режимы:
- **Interactive Server** — компонент выполняется на сервере
- **Interactive WebAssembly** — компонент выполняется в браузере
- **Interactive Auto** — первый запрос на сервере, затем загрузка WASM
- **Static SSR** — серверный рендеринг без интерактивности (аналог MVC Razor Pages)

### Шаг 3: Реализуй гибридный подход (опционально)

Blazor и MVC могут сосуществовать в одном проекте:
- MVC для основных страниц и SEO
- Blazor для интерактивных виджетов

## Инструменты и подходы

| Инструмент | Назначение |
|---|---|
| SignalR | Real-time соединение клиент ↔ сервер |
| WebAssembly (WASM) | Выполнение .NET в браузере |
| Razor-компоненты | Единица UI в Blazor (.razor файлы) |
| ASP.NET Core Routing | Маршрутизация в обоих подходах |
| Dependency Injection | Внедрение зависимостей |

## Ограничения и типичные ошибки

1. **Blazor Server не для high-load:** Постоянное соединение через SignalR создаёт нагрузку на сервер. Для тысяч одновременных клиентов это может стать узким местом.

2. **Blazor WASM — декомпиляция:** Клиентские DLL можно легко декомпилировать. Не храни бизнес-логику, API-ключи или секреты в клиентском коде.

3. **Первый запуск WASM медленный:** Загрузка .NET Runtime + DLL может занимать несколько секунд. Используй кэширование и прелоадинг.

4. **MVC не мёртв:** Blazor дополняет, а не заменяет MVC. Для классических сайтов, блогов, лендингов MVC + Razor Pages остаётся лучшим выбором.

5. **SEO Blazor ограничен:** Blazor Server рендерится на сервере (хорошо для SEO), но Blazor WASM — полностью клиентский (плохо для SEO). В .NET 8+ добавлен SSR режим.

6. **JavaScript всё ещё нужен:** Для некоторых задач (работа с Canvas, WebGL, сторонние библиотеки) потребуется JavaScript interop в Blazor.
